# Created: 2026-04-16

## REPRODUCIBILITY BOUNDING ANALYSIS

### EXPECTATION MANAGEMENT

The paper correctly distinguishes between three types of replication:

1. **ORDERING** (primary goal for most experiments)
   - "Reproduce the backend ranking: sync wins at small scale, io_uring wins at large scale"
   - Evidence: Tables 1-2 show consistent ordering
   - Appropriate for SafeTensors matrix

2. **REGIME BEHAVIOR** (secondary goal)  
   - "Reproduce crossover at the right scale threshold"
   - Evidence: Qwen3-4B-8B band shows default selector decisions
   - Appropriate for default policy evaluation

3. **EXACT TIMING** (NOT claimed - NOT required)
   - Paper explicitly says NOT to chase identical milliseconds
   - Appendix line 43-44: "prioritise ordering and regime behaviour"
   - Root README line 25: "Match backend ordering...not byte-identical timings"

### NEW READER HIDDEN ASSUMPTIONS CHECK

What a new reader might miss WITHOUT the full paper+repo combination:

1. **Root requirement**: "Requires root for /proc/sys/vm/drop_caches"
   - Hidden in: experimental_setup.tex line 81-82
   - Impact: Cold cache replication impossible without this
   - OK: Documented but could be more prominent

2. **Dedicated machine requirement**: "assumed a dedicated host"
   - Hidden in: experimental_setup.tex line 24 and appendix line 49
   - Impact: Shared machine cache drops disrupt other tenants
   - OK: Documented appropriately

3. **Linux-only requirement**: "io_uring backend is Linux-only"
   - Hidden in: experimental_setup.tex line 30 ("I/O experiments are Linux-only")
   - OK: Explicitly stated

4. **GPU requirement for vLLM**: "GPU only for vLLM/GPU-backed paths"
   - Hidden in: README.md line 45
   - OK: Documented

5. **Model fetch behavior**: "downloads via Hub if needed"
   - Implicit in: profile binary behavior, not explicit docs

- Exact timing replication NOT expected
- Ordering replication IS expected
