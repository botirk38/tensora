\subsection{Headline findings}

Three patterns emerge from the Rust matrices. First, SafeTensors loading consistently favours async over sync, with the advantage growing from 1.13$\times$ at 0.6B to 2.40$\times$ at 32B. Second, ServerlessLLM-style partitioning similarly favours async at every scale; sync never wins when the read geometry is range-dominated. Third, the adaptive default backend matches or approaches the best explicit backend throughout, validating that the regime structure is predictable enough for automatic selection.

\subsection{Rust Backend Results}
\label{sec:rust-results}

The Rust harness is the authoritative backend-attribution layer because it measures checkpoint loading directly, without Python tensor conversion or serving-engine initialization costs. All cells report medians from five cold-cache repetitions.

The format determines both the baseline cost and how much backend choice can improve it.

Tensora's async backend is 1.6--13.8$\times$ faster than hf-native across all scales. These results confirm that Tensora's Rust-based approach provides meaningful improvement over the standard Python loading path, independent of backend choice.

\subsection{Layer summary}

Four principles emerge. (1) Backend ranking depends on checkpoint format: async outperforms sync on both SafeTensors and ServerlessLLM, with the advantage growing with model scale. (2) The magnitude of advantage differs by format: SafeTensors shows larger speedups (up to 2.40$\times$) because its contiguous read pattern benefits more from async's concurrent scheduling at scale. (3) The adaptive default backend matches or approaches the best explicit backend in most cells, validating that format-conditioned backend selection is both effective and automatable. (4) Tensora's async backend is 1.6--13.8$\times$ faster than the hf-native baseline, confirming that the framework provides practical improvement over stock Python-level loading.
