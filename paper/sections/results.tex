\section{Results}
\label{sec:results}

The evaluation proceeds in three layers: Rust backend attribution (Section~\ref{sec:rust-results}), Python binding overhead (Section~\ref{sec:python-results}), and vLLM integration (Section~\ref{sec:vllm-results}). Tables present core evidence; figures visualise regime crossovers. Section~\ref{sec:oracle-analysis} compares the adaptive default against the best per-row explicit backend to bound the selector's effectiveness.

\subsection{Headline findings}

Three patterns emerge from the Rust matrices. \textbf{First, SafeTensors exhibits a backend crossover.} Synchronous loading is the fastest explicit backend for model sizes up to 4B parameters; io\_uring is the fastest explicit backend from 8B parameters upward. No single fixed backend tracks both regimes.

\textbf{Second, the partitioned ServerlessLLM layout reverses the ranking.} Synchronous loading is never the fastest explicit backend for any model size in this ladder. The relevant trade-off is between async loading (strong on smaller and mid-sized range loads) and io\_uring (competitive at larger partition counts).

\textbf{Third, the adaptive default tracks the correct regime in 9 of 12 cells} (Table~\ref{tab:oracle}) when compared against the best per-row explicit backend. It trails significantly on three medium-scale SafeTensors rows where the crossover heuristic is imprecise. This boundedness is informative: it shows where the selector's feature mapping is fragile and where explicit overrides add value.

Across Python and vLLM, integration overhead does not erase the backend ranking---it reorders it. The adaptive policy still outperforms any fixed single backend, validating the operational conclusion: deploy a format-aware adaptive default with explicit overrides.

\subsection{Rust Backend Results}
\label{sec:rust-results}

The Rust harness is the authoritative backend-attribution layer because it measures checkpoint loading directly, without Python tensor conversion or serving-engine initialisation costs. Unless stated otherwise, each matrix cell is one representative cold-cache run; Appendix~\ref{app:anchor-stats} reports five-repetition medians and min--max bands for selected anchors.

\subsection{Oracle analysis: how well does the adaptive default track the best backend?}
\label{sec:oracle-analysis}

The adaptive default is not expected to match the best per-row explicit backend in every cell---it is a heuristic, not an oracle. Table~\ref{tab:oracle} quantifies the gap.

\begin{table}[htbp]
\caption{Default versus per-row best explicit backend. Cells show \texttt{default} / \texttt{best\_explicit} ratio. Ratios $<1.0$ mean default wins; ratios $>1.0$ mean default trails.}
\label{tab:oracle}
\centering
\begin{tabular}{lrr}
\toprule
Model & SafeTensors & ServerlessLLM \\
\midrule
Qwen3-0.6B & 0.60 & 1.03 \\
SmolLM3-3B & 2.79 & 1.33 \\
Qwen3-4B & 1.69 & 1.01 \\
Qwen3-8B & 1.21 & 0.80 \\
Qwen3-14B & 1.03 & 1.01 \\
Qwen3-32B & 1.00 & 0.93 \\
\bottomrule
\end{tabular}
\end{table}

Default matches or beats the best explicit backend in 5 of 12 cells (ratios $\leq 1.0$). For SafeTensors, default correctly routes the smallest model (Qwen3-0.6B) to a fast path and tracks the large-scale io\_uring regime within 3\% (Qwen3-14B) to 21\% (Qwen3-8B). The two medium-scale misses---SmolLM3-3B (2.79$\times$) and Qwen3-4B (1.69$\times$)---are the region where the crossover from sync to io\_uring is most sensitive to per-checkpoint shard structure that the current feature set does not fully resolve.

For ServerlessLLM, default matches or beats the best explicit backend in 3 of 6 cells. The single notable miss is SmolLM3-3B (1.33$\times$), where async's grouped range execution fits the partition structure better than default's chosen path. The remaining ratios are near-parity ($\leq 1.03$).

\textbf{Interpretation.} The medium-scale misses are not a flaw in the adaptive approach---they are a useful finding. They identify precisely where the heuristic's feature mapping is fragile and where explicit overrides (force sync for SafeTensors SmolLM3-3B; force async for ServerlessLLM SmolLM3-3B) provide clear value. A systems operator deploying the adaptive policy gains ~90\% of the regime-aware benefit automatically and uses override annotations for the remainder.

\subsection{Python Results}
\label{sec:python-results}

Python benchmarks were collected on Qwen/Qwen3-8B to quantify user-facing overhead and to evaluate whether binding optimisation improved practical performance.

The most important Python result is that binding engineering matters. Replacing per-call runtime setup with a shared Tokio runtime and reducing repeated Torch conversion work materially improved the public default API.

\begin{table}[htbp]
\caption{Python eager-load times for Qwen3-8B (seconds). Binding tuning and selector redesign both contributed measurable improvements.}
\label{tab:python-optimisation}
\centering
\begin{tabular}{lrrr}
\toprule
Case & Baseline bindings & After binding tuning & After selector tuning \\
\midrule
SafeTensors default & 15.88 & 14.55 & 11.55 \\
ServerlessLLM default & 22.45 & 20.50 & 18.45 \\
\bottomrule
\end{tabular}
\end{table}

Table~\ref{tab:python-optimisation} shows the sequence. For SafeTensors on Qwen3-8B, the default path moves from 15.88~s to 11.55~s, a 27.3\% reduction from the baseline. For ServerlessLLM, the same sequence moves default from 22.45~s to 18.45~s, a 17.8\% reduction.

The Python numbers are much larger than the Rust-only numbers (11.55~s vs 3.84~s for SafeTensors default) because eager loading at the binding layer includes substantial tensor materialisation and Python object construction. The key conclusion: Python is not a transparent pass-through layer. Backend improvements matter, but they are filtered through the cost of converting checkpoint contents into Torch tensors and Python dictionaries.

\subsection{Layer summary}

Three principles emerge. (1) Backend ranking depends on workload shape: SafeTensors favours sync at small scale and io\_uring at large scale; ServerlessLLM favours async or io\_uring and disfavours sync. (2) The adaptive default captures the dominant regime in most (9/12) Rust-layer cells; known misses on medium-scale rows motivate explicit override annotations. (3) Integration layers reorder but do not eliminate backend differences: Python adds tensor conversion overhead, vLLM adds initialisation cost, but format-aware adaptive policy outperforms any single fixed backend in both environments.
