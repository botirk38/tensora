\section{Experimental Setup}
\label{sec:experimental_setup}

The evaluation is designed to separate mechanism from deployment impact. A Rust benchmark isolates backend execution. A Python benchmark measures the public binding surface. A vLLM benchmark measures the serving-engine boundary. The same timing number should not be interpreted the same way across all three layers.

\subsection{Measurement layers}

\begin{enumerate}
\item \textbf{Rust backend attribution.} Measures checkpoint reading, format handling, and tensor assembly through a selected backend. It excludes Python interpreter costs and serving-engine GPU initialization. This layer answers which submission policy is fastest for a fixed logical read plan.
\item \textbf{Python public API.} Measures what a Python client observes: FFI transitions, tensor construction, object materialization, and binding overhead in addition to the backend path. This layer answers whether Rust-level differences survive the library interface.
\item \textbf{vLLM integration.} Measures loading inside a serving engine, including CUDA context setup, memory pools, runtime configuration, model initialization, and time-to-first-token when requested. This layer answers whether loader choice matters in an end-to-end serving context.
\end{enumerate}

These layers are deliberately not collapsed. A Rust result provides backend evidence; a vLLM result provides deployment evidence. The limitations section states where claims cannot move across layers.

\subsection{Research questions}

The study answers five questions.
\begin{itemize}
\item \textbf{RQ1: Backend attribution.} How much does backend choice affect checkpoint loading for real LLM checkpoints?
\item \textbf{RQ2: Layout dependence.} Does changing checkpoint layout change the backend ranking?
\item \textbf{RQ3: API overhead.} How much does the Python binding layer add, and can binding design reduce it?
\item \textbf{RQ4: Serving impact.} Do loader effects survive inside vLLM initialization and TTFT?
\item \textbf{RQ5: Policy.} What should an operational default promise, and where should explicit overrides remain available?
\end{itemize}

RQ1--RQ2 use the Rust matrices, RQ3 uses Python benchmarks, RQ4 uses vLLM, and RQ5 combines all layers.

\subsection{Backends, layouts, and model ladder}

The Rust layer evaluates four eager-loading policies:
\begin{itemize}
\item \textbf{sync}: blocking POSIX reads with dynamic chunking and host-side parallelism,
\item \textbf{async}: Tokio-based loading with dynamic batching and per-file grouping,
\item \textbf{io\_uring}: Linux \texttt{io-uring} with persistent rings and workload-driven planning,
\item \textbf{default}: the adaptive selector and planner described in Section~\ref{sec:design}.
\end{itemize}

The two layouts are \textbf{SafeTensors}, a whole-file or multi-shard contiguous layout, and \textbf{ServerlessLLM-style partitioning}, an indexed layout accessed through grouped range reads. Memory-mapped APIs are exposed for lazy Python access but are not treated as eager cold-load backends in the Rust matrix.

The model ladder contains six public checkpoints: \texttt{Qwen/Qwen3-0.6B}, \texttt{HuggingFaceTB/SmolLM3-3B}, \texttt{Qwen/Qwen3-4B}, \texttt{Qwen/Qwen3-8B}, \texttt{Qwen/Qwen3-14B}, and \texttt{Qwen/Qwen3-32B}. This ladder spans small, medium, and large checkpoints and includes a non-Qwen model to reduce overfitting to one family. A single model size could justify only a constant backend choice; the ladder is what exposes crossovers.

\subsection{Cache state and comparability}

Cold-cache behavior is the primary condition for Rust and Python eager-load measurements. A cold-cache run performs:
\begin{enumerate}
\item \texttt{sync},
\item \texttt{echo 3 > /proc/sys/vm/drop\_caches},
\item benchmark invocation.
\end{enumerate}
Warm-cache conditions skip the cache drop and reuse page-cache state from prior reads. On restricted containers where direct cache dropping is unavailable, the repository-supported page-cache eviction path is recorded as a methodological deviation.

Each backend receives the same logical tensor read plan for a given model and layout. Differences in wall time come from submission strategy, parallelism, chunking, coalescing, and buffer management. If \texttt{default} loses on a row, that is evidence about the selector and planner under the same workload constraints, not a different task.

\subsection{Baselines and reporting}

The Rust matrices compare Tensora backends against one another because the purpose is backend attribution under identical format handling. The vLLM matrix includes a deployment-style baseline: \texttt{native} is vLLM's stock loading path, while loaders prefixed \texttt{ts\_} use Tensora-backed integrations through the same harness.

Full matrices report representative cold-cache times for readability. Five-repetition anchor sets validate selected small, medium, and large regimes and are reported in Appendix~\ref{app:anchor-stats}. We treat small single-cell gaps cautiously unless supported by repeated anchors or consistent ordering across rows. The main claims rely on large effect sizes and repeated regime patterns: sync versus \texttt{io\_uring} at H100 SafeTensors scale, sync's failure on ServerlessLLM, and cross-environment changes in large-scale winners.

\subsection{Falsification criteria}

The central claim would be weakened if one fixed explicit backend dominated both layouts and all scales, if sync remained fastest for the largest H100 SafeTensors checkpoints, or if sync became the strongest backend across the ServerlessLLM matrix. It would be refined, not falsified, if another host showed the same regime changes at different model sizes.

The adaptive-policy claim would be weakened if \texttt{default} systematically lost to one explicit backend across the full matrix. The observed misses are narrower: medium-scale SafeTensors and selected partitioned rows. They motivate explicit overrides rather than a return to a global fixed backend.
