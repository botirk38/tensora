\section{Experimental Setup}
\label{sec:experimental_setup}

The evaluation is designed to measure how checkpoint format and model scale affect I/O backend performance. A Rust benchmark isolates backend execution; a vLLM benchmark measures serving-engine initialization; an external baseline compares against upstream loading.

\subsection{Measurement layers}

\begin{enumerate}
\item \textbf{Rust backend attribution.} Measures checkpoint reading, format handling, and tensor assembly through a selected backend. It excludes Python interpreter costs and serving-engine GPU initialization. This layer answers which submission mechanism is fastest for a fixed logical read plan.
\item \textbf{vLLM integration.} Measures loading inside a serving engine, including CUDA context setup, memory pools, runtime configuration, and model initialization. This layer answers whether loader choice matters in an end-to-end serving context.
\end{enumerate}

These layers are deliberately not collapsed. A Rust result provides backend evidence; a vLLM result provides deployment evidence.

\subsection{Research questions}

The study answers three questions.
\begin{itemize}
\item \textbf{RQ1: Format dependence.} Does changing checkpoint format change which I/O backend is fastest?
\item \textbf{RQ2: Scale interaction.} How does model scale interact with format--backend combinations?
\item \textbf{RQ3: Integration transfer.} Does backend advantage survive inside vLLM initialization, and does Tensora outperform the upstream baseline?
\end{itemize}

RQ1 and RQ2 use the Rust matrices. RQ3 uses the vLLM integration benchmarks and the hf-native baseline.

\subsection{Platform}

All primary measurements were collected on Modal H100 containers (Table~\ref{tab:platform}). Each container provisions an NVIDIA H100 GPU with 32\,GB RAM and 512\,GB ephemeral NVMe. The container runs gVisor (a userspace kernel), which blocks \texttt{io\_uring} with \texttt{ENOSYS}; only sync and async backends are available. Models are downloaded from the HuggingFace CDN on each cold run, and page-cache eviction uses \texttt{posix\_fadvise(DONTNEED)}.

\subsection{Backends, layouts, and model ladder}

The Rust layer evaluates two explicit I/O backends:
\begin{itemize}
\item \textbf{sync}: blocking POSIX reads with dynamic chunking and host-side parallelism,
\item \textbf{async}: Tokio-based loading with dynamic batching and per-file grouping.
\end{itemize}

The two layouts are \textbf{SafeTensors}, a whole-file or multi-shard contiguous layout, and \textbf{ServerlessLLM-style partitioning}\footnote{This paper evaluates a partitioned range layout inspired by the ServerlessLLM checkpoint format~\cite{serverlessllm_repo}. The conversion procedure produces index files and partition files matching ServerlessLLM's structure, but the benchmarks measure Tensora's own read-planning and backend layers against this layout, not the ServerlessLLM system's tiered loading pipeline (Section~\ref{sec:related_work}).}, an indexed layout accessed through grouped range reads. Memory-mapped APIs are exposed for lazy Python access but are not treated as eager cold-load backends in the Rust matrix.

The model ladder contains six public checkpoints: \texttt{Qwen/Qwen3-0.6B}, \texttt{HuggingFaceTB/SmolLM3-3B}, \texttt{Qwen/Qwen3-4B}, \texttt{Qwen/Qwen3-8B}, \texttt{Qwen/Qwen3-14B}, and \texttt{Qwen/Qwen3-32B}. This ladder spans small, medium, and large checkpoints and includes a non-Qwen model to reduce overfitting to one family.

\subsection{External baseline}

The \textbf{hf-native} baseline uses the upstream Hugging Face \texttt{safetensors} crate's \texttt{std::fs::read}-based loading path. This represents the default loading strategy available to any SafeTensors consumer without Tensora's backend-aware read planning. By comparing Tensora's backends against this baseline, we isolate the value of explicit backend selection over the standard upstream path.

\subsection{Cache state and comparability}

Cold-cache behavior is the primary condition for all Rust eager-load measurements. A cold-cache run performs page-cache eviction via \texttt{posix\_fadvise(DONTNEED)} followed by the benchmark invocation.

Each backend receives the same logical tensor read plan for a given model and layout. Differences in wall time come from submission strategy, parallelism, chunking, coalescing, and buffer management.

\subsection{Statistical methodology}
\label{sec:baselines-reporting}

All cells in the primary Rust matrix are backed by five cold-cache repetitions (180 runs across 36~cells). Medians and ranges are reported for all cells.

Where anchor ranges overlap between backends, we avoid forcing a strict ranking and instead state that the difference is not statistically separable at the resolution of five cold-cache repetitions. Separability is assessed by range overlap: if the ranges of two backends overlap, neither can be declared confidently faster. The main claims rely on large effect sizes and consistent regime patterns (e.g., async advantage on SafeTensors, sync-never-wins on ServerlessLLM), not on close-call single-cell comparisons.

\subsection{Falsification criteria}

The central claim would be weakened if a single fixed backend dominated both layouts and all scales, if sync outperformed async at the largest SafeTensors checkpoint, or if sync became the strongest backend across the ServerlessLLM matrix.
