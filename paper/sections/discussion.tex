\section{Discussion}
\label{sec:discussion}

The central lesson is simple: checkpoint loading is a workload-shape problem. Layout determines request geometry; scale determines how much queue depth and parallelism can be used; deployment context determines which kernel interfaces are available; integration layers determine how much backend improvement reaches the user.

\subsection{Why the layouts diverge}

SafeTensors and ServerlessLLM-style partitioning expose different I/O problems. SafeTensors is mostly whole-file or multi-shard eager loading. For small checkpoints, a practical sync implementation is hard to beat: it has low orchestration overhead and enough parallelism for the available work. As checkpoint size and shard count grow, keeping more I/O in flight matters, and the H100 matrix shifts toward \texttt{io\_uring}.

ServerlessLLM-style partitioning is different. The loader reconstructs tensors from partition files and range-oriented access. This makes grouping, coalescing, and batching central. In this regime the sync-first intuition from SafeTensors fails: sync is never the fastest backend in the H100 ServerlessLLM matrix. The best policy is therefore not ``use \texttt{io\_uring}'' or ``use sync''; it is ``read the format and scale before choosing.''

\subsection{Why integration changes the ranking}

Rust results answer the backend-attribution question. Python and vLLM answer different questions. Python adds interpreter overhead, tensor conversion, and object materialisation. vLLM adds CUDA context setup, memory pools, runtime configuration, and generation setup. These costs can dilute or reorder pure I/O differences.

That reordering is not a weakness in the evaluation; it is the deployment reality. A loader that wins only in a microbenchmark but disappears at the serving boundary is less valuable. Conversely, a loader that saves tens of seconds in vLLM initialisation is operationally meaningful even if the exact Rust ranking changes.

\subsection{What the adaptive policy should promise}

The adaptive selector should promise three things and no more. First, it should avoid obviously wrong global defaults, such as sync for all ServerlessLLM loads or \texttt{io\_uring} for every small SafeTensors load. Second, it should degrade gracefully when an interface is unavailable, as in H200 and A100 containers where \texttt{io\_uring} ring creation is blocked. Third, it should expose explicit overrides for boundary cases.

It should not promise universal optimality. Table~\ref{tab:oracle} shows medium-scale misses, and those misses are useful: they reveal which features are insufficient. A production policy can incorporate more signals over time, but the paper's evidence already rules out a single fixed backend.

\subsection{External validity and replication}

Absolute milliseconds are the least portable part of the study. Ordering regimes are more portable. The H200 and A100 results support this distinction: small SafeTensors loads favour sync across all environments, while the mid-to-large-scale winner changes with storage topology and \texttt{io\_uring} availability. A replication should therefore ask whether the same regime changes appear, not whether a different host reproduces every number.

\subsection{Design implications}

For practitioners, the implication is to log and select on workload features: format family, total bytes, shard or partition count, backend availability, and integration layer. For researchers, the implication is to report loading results as layered measurements. A Rust microbenchmark, a Python API benchmark, and a serving-engine cold-start benchmark are all useful, but they are not interchangeable constructs.
