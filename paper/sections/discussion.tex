\section{Discussion}
\label{sec:discussion}

The main result is not that one backend wins. The main result is that a checkpoint loader must treat layout and scale as policy inputs.

\subsection{Mechanism: why layout changes the answer}

SafeTensors and ServerlessLLM-style partitioning create different I/O problems. SafeTensors exposes large contiguous reads after metadata parsing. For small checkpoints, a practical sync backend has low overhead and enough parallelism; additional scheduling machinery does not amortize. As size and shard count grow, the H100 workload has enough in-flight work for \texttt{io\_uring} to pay off.

ServerlessLLM-style partitioning changes the request geometry. The loader reconstructs tensors from indexed partition files, so grouping, coalescing, and range batching dominate. The SafeTensors rule ``sync can be good for small loads'' does not transfer: sync never wins in the H100 ServerlessLLM matrix. The format, not the API name, explains the reversal.

\subsection{Integration: why rankings move across layers}

Rust, Python, and vLLM answer different questions. Rust isolates backend mechanics. Python measures a public library boundary with FFI, tensor conversion, and object construction. vLLM measures serving startup with CUDA context setup, memory pools, runtime configuration, and generation setup. These costs can dilute a pure I/O advantage or reorder two close backends.

This is not a flaw in the methodology; it is the deployment reality. A systems paper about loading should show both mechanism and propagation. Tensora's gains matter because they remain visible in vLLM cold initialization even after fixed engine costs are included.

\subsection{Policy: what \texttt{default} should promise}

An adaptive default should promise three bounded properties. First, it should avoid clearly wrong global choices: sync for all partitioned loads, or \texttt{io\_uring} for every tiny contiguous load. Second, it should degrade gracefully when a backend is unavailable, as in containers that block \texttt{io\_uring}. Third, it should expose explicit overrides for known boundary cases.

It should not promise oracle optimality. Table~\ref{tab:oracle} shows misses where the current feature set is too coarse. Those misses are useful because they identify where the next policy should add signals: finer shard geometry, storage topology, or learned per-environment calibration. The correct engineering response is transparency plus overrides, not hiding the selector behind a single API name.

\subsection{What transfers}

Absolute milliseconds are machine-specific. Ordering regimes transfer more plausibly. In this study, small SafeTensors loads favor sync across H100, H200, and A100. The large-scale winner does not transfer by name: H100 favors \texttt{io\_uring}, H200 favors async when \texttt{io\_uring} is blocked, and A100 shows a non-monotonic recovery at the largest SafeTensors row. A replication should therefore ask whether workload-shape changes persist, not whether every number matches.

\subsection{Implications}

For practitioners, log the features that determine loader choice: format, bytes, shard or partition count, backend availability, cache condition, and integration layer. Expose a default, but also expose explicit backend labels and override hooks.

For researchers, report loading results as layered constructs. A Rust microbenchmark, a Python API benchmark, and a serving-engine startup benchmark are all valuable; none subsumes the others. A paper that reports only one layer risks explaining the wrong phenomenon.
