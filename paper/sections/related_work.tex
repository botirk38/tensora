\section{Related Work}
\label{sec:related_work}

Tensora sits between four lines of work: LLM serving engines, serverless cold-start systems, checkpoint formats and loading infrastructure, and operating-system I/O policy. Prior work either optimizes the serving pipeline after weights are resident, avoids cold loads through placement and caching, changes the checkpoint/storage substrate, or improves kernel-level page service. Tensora targets the application-level loading decision: given a concrete checkpoint layout and host capability set, which userspace backend should execute the read plan?

\subsection{LLM serving systems}

Modern serving systems primarily optimize inference after weights are resident. vLLM introduces PagedAttention for KV-cache management and high-throughput serving \cite{pagedattention}. Orca uses iteration-level scheduling \cite{orca}. Sarathi-Serve studies the prefill--decode scheduling trade-off \cite{sarathi_serve}. DeepSpeed Inference applies model parallelism and kernel optimizations \cite{deepspeed_inference_paper}. These systems make the steady-state serving pipeline efficient, but typically treat checkpoint loading as a fixed initialization cost.

Our vLLM integration is therefore a boundary test. Rust measurements explain backend mechanics; vLLM measurements test whether those mechanics survive in an engine with CUDA setup, memory pools, runtime configuration, and generation state. The contribution is not a new serving scheduler. It is a loader policy whose deployment impact can be observed inside a serving engine.

\subsection{Serverless and cold-start mitigation}

Serverless LLM systems make cold start a first-order concern because evicted models must be restored on demand. ServerlessLLM \cite{FU:2024} proposes multi-tier loading across GPU, CPU memory, and storage. HydraServe \cite{hydraserve} reduces cold-start latency in public-cloud serving through placement and pipeline parallelism. Snapshot-based warm-up systems \cite{silva2021snapshot} pre-materialize process state to bypass portions of startup.

These systems optimize \emph{whether}, \emph{where}, or \emph{when} loading occurs. Tensora asks the complementary question that remains when a load is unavoidable: how should the required reads be submitted? The answer matters in any tiered design because each tier still presents a read plan to userspace.

\subsection{Checkpoint formats and loading systems}

Checkpoint formats define the request geometry visible to a loader. SafeTensors \cite{safetensors_docs} stores tensor bytes in safe, contiguous files or shards. GGUF targets memory-mapped CPU inference and quantized deployments. ServerlessLLM's layout \cite{serverlessllm_repo} distributes tensors across partition files with a central index, enabling tiered or partial placement at the cost of more range-oriented access.

Recent systems work attacks loading from other layers. BlitzScale \cite{blitzscale} shares already-loaded models across inference instances over high-speed interconnects. PPC/MAIO \cite{fast26_maio} uses a programmable kernel page cache to adapt prefetching and eviction to LLM I/O patterns. MAIO operates below the system-call interface; Tensora operates above it. The two approaches are complementary: kernel cache policy changes how pages are served, while userspace backend policy changes how reads are issued.

A separate characterization line studies checkpoint/restore I/O patterns on parallel file systems \cite{arxiv2512_checkpoint_io}, including buffered versus direct paths and liburing. That work focuses on training checkpointing in HPC environments. Tensora focuses on inference-time checkpoint loading in single-node serving and follows the effects through Python and vLLM.

\subsection{I/O backend policy}

The Linux I/O interface landscape includes blocking POSIX calls, async runtimes layered above kernel primitives, POSIX AIO, and \texttt{io\_uring}. Database research provides the closest methodological warning: Jasny et al. \cite{IOURING_DATABASE_GUIDELINES} show that \texttt{io\_uring} is not automatically faster. Its benefit depends on request shape, queue depth, batching, and implementation details. Bulk sequential reads can already amortize syscall overhead; fragmented or high-concurrency reads can benefit from batched submission.

LLM checkpoint loading spans both regimes. SafeTensors resembles contiguous bulk transfer with shard-level fanout. ServerlessLLM-style partitioning resembles grouped range access. Tensora translates the workload-shape lesson to LLM loading and adds three elements missing from prior I/O studies: format-conditioned comparison, cross-layer measurement, and capability-gated fallback when \texttt{io\_uring} is unavailable in containerized environments.

\subsection{Positioning}

We do not introduce a checkpoint format, a serving scheduler, or a kernel cache. We measure the load mechanics between them: four backend policies, two layouts, six public checkpoints, three measurement layers, and three deployment environments. The contribution is operational and falsifiable: backend choice depends jointly on layout, scale, and runtime capability; therefore a production loader should select from measured workload features and expose explicit overrides.
