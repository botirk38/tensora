\section{Limitations}
\label{sec:limitations}

The paper makes a bounded systems claim: backend choice should depend on workload shape. It does not prove a universal selector, universal thresholds, or universal superiority of any I/O API.

The primary matrices come from one H100 host with local NVMe and working \texttt{io\_uring}. H200 and A100 provide useful but different validation: network-storage containers where \texttt{io\_uring} is blocked. They test regime transfer, but they are not three identical replicas of the same hardware/storage stack. The model ladder covers six public checkpoints from 0.6B to 32B parameters and two layouts. It does not cover GGUF, quantized-only deployments, object stores, distributed filesystems, RAID variants, direct-I/O configurations, Windows/macOS backends, or multi-node distributed loading. ``Load time'' also means different things at different layers: Rust timings approximate backend-attributed disk-to-tensor work; vLLM timings add Python integration overhead, engine initialization, and CUDA setup. A Rust result should not be quoted as a serving result.

The results compare three specific implementations (sync POSIX, Tokio async, and \texttt{io\_uring}), not the best possible implementation of each API. A different sync backend with adaptive parallelism or a differently tuned \texttt{io\_uring} setup might shift the crossover points. This does not invalidate the central claim, that checkpoint loading has identifiable regimes that shift with layout and scale, because that claim depends on the workload-shape explanation (contiguous vs.\ partitioned reads), not on any single backend being universally optimal.

Half of the H100 Rust matrix (18 of 36 cells) is backed by five-repetition anchor sets; the remaining cells are representative cold-cache runs. Repeating the full H100 matrix for complete variance estimation would strengthen the statistical base; the Modal repeated anchor matrix (five repetitions across all six models and both formats) provides the closest available proxy. The paper's main claims rest on separable regime distinctions (SafeTensors sync-to-\texttt{io\_uring} crossover, ServerlessLLM sync-never-wins), confirmed by non-overlapping anchor ranges. Close-call rankings are explicitly flagged as not statistically separable. No explicit backend dominates across layout, scale, and environment. Held-out validation on Mistral-7B-Instruct confirms regime transfer but also shows that effect sizes vary with shard geometry; broader held-out coverage would further test generalization.

The Python integration layer is not independently benchmarked in this study. The Rust-to-vLLM stack provides end-to-end evidence that backend effects propagate through the serving stack, but the Python-level contribution is not separately quantified.
