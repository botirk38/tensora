\section{Limitations}
\label{sec:limitations}

The results support regime claims in a controlled harness; this section states what does \emph{not} follow without further evidence.

\subsection{External validity}

All measurements come from one H100-based Linux host with local NVMe. Ordering regimes (which backend wins at which scale and layout) are more likely to transfer across machines than absolute timings, but that transfer is itself an empirical claim. Different NVMe generations, RAID or network-attached storage, and filesystem configurations would test the stability of the observed crossovers. A second measurement site is the strongest check \cite{IOURING_DATABASE_GUIDELINES}.

The six-model ladder covers 0.6B to 32B parameters across two formats. It does not exhaust the design space of checkpoint formats (e.g., GGUF), model families, or storage hardware.

\subsection{Construct validity}

Each measurement layer operationalizes ``loading time'' differently. Rust timings approximate pure I/O and deserialization; Python timings add tensor materialization and FFI overhead; vLLM timings include GPU bring-up and memory pool allocation. None is the same construct as user-perceived cold-start latency in production, which adds network RTT, batching delays, and queuing. Section~\ref{sec:experimental_setup} makes these definitions explicit to prevent claim smuggling across constructs.

\subsection{Internal validity and selector boundedness}

The adaptive default does not match the best per-row explicit backend across all cells. On SafeTensors, it is 2.8$\times$ behind explicit sync for SmolLM3-3B and 1.7$\times$ for Qwen3-4B---the medium-scale crossover region where the heuristic's feature mapping is imprecise. On ServerlessLLM, it lags explicit async by 1.33$\times$ on SmolLM3-3B. These misses are documented in Section~\ref{sec:oracle-analysis} and inform the operational recommendation: deploy explicit overrides for the medium-scale region rather than tuning the heuristic to chase edge cases.

The full Rust and vLLM matrices report representative cold-cache times. Anchor repetitions (Appendix~\ref{app:anchor-stats}) validate regime stability for selected cells, but per-cell variance is omitted from main tables for readability. Gaps below ~10\% in individual cells should be read cautiously.

\subsection{Scope of claims}

The evidence supports the adaptive policy as an \textbf{engineering heuristic justified by measurement}, not a claim of universal optimality. The selector automates stable regime patterns (sync-for-small, io\_uring-for-large SafeTensors; async-or-io\_uring for ServerlessLLM) while acknowledging known misses on medium-scale rows. The falsification conditions from Section~\ref{sec:experimental_setup} apply: the adaptive heuristic would be undermined if sync won at the largest SafeTensors scales or if sync became competitive on ServerlessLLM; neither is observed in this harness.

Future work suggested by these limitations is practical rather than speculative: cross-hardware replication, filesystem and mount-option sweeps, additional formats (GGUF, Direct I/O variants), and learned or calibrated selector tuning would refine boundary stability without contradicting the structural claim that scale and layout change the best policy.
