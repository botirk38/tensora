\section{Limitations}
\label{sec:limitations}

The paper makes a bounded systems claim: backend policy should depend on workload shape. It does not prove a universal selector, universal thresholds, or universal superiority of any I/O API.

\subsection{External validity}

The primary matrices come from one H100 host with local NVMe and working \texttt{io\_uring}. H200 and A100 provide useful but different validation: network-storage containers where \texttt{io\_uring} is blocked. They test fallback behavior and regime transfer, but they are not three identical replicas of the same hardware/storage stack.

The model ladder covers six public checkpoints from 0.6B to 32B parameters and two layouts. It does not cover GGUF, quantized-only deployments, object stores, distributed filesystems, RAID variants, direct-I/O configurations, Windows/macOS backends, or multi-node distributed loading.

\subsection{Construct validity}

``Load time'' means different things at different layers. Rust timings approximate backend-attributed disk-to-tensor work. Python timings add interpreter and binding costs. vLLM timings add engine initialization, CUDA setup, memory pools, and generation state. A Rust result should not be quoted as a serving result, and a vLLM result should not be used to rank pure I/O APIs without qualification.

\subsection{Statistical limits}

Half of the H100 Rust matrix (24 of 48 cells) is backed by five-repetition anchor sets; the remaining cells are representative cold-cache runs. Appendix~\ref{app:anchor-stats} shows which anchored rankings are statistically separable (non-overlapping ranges) and which overlap. The paper's main claims rest on separable regime distinctions (SafeTensors sync-to-io\_uring crossover, ServerlessLLM sync-never-wins), which are confirmed by non-overlapping anchor ranges. Close-call rankings (e.g., default vs.\ sync at SafeTensors 0.6B and 8B; async vs.\ default vs.\ io\_uring at all ServerlessLLM scales) are explicitly flagged as not statistically separable. Differences below roughly 5\% at unanchored scales should be read cautiously unless supported by cross-row consistency.

Three unanchored model sizes (SmolLM3-3B, Qwen3-4B, Qwen3-14B) contain cells where the top two backends are within 3\%; targeted five-repetition anchoring at these cells would clarify borderline rankings.

\subsection{Selector boundedness}

The adaptive default is not an oracle. On H100 SafeTensors it trails explicit sync by 2.79$\times$ for SmolLM3-3B and 1.69$\times$ for Qwen3-4B, and trails explicit \texttt{io\_uring} by 1.21$\times$ for Qwen3-8B. On H100 ServerlessLLM it trails explicit async by 1.33$\times$ for SmolLM3-3B. On A100, it misses the mid-range async regime for SafeTensors.

These failures are part of the result. They show that a deployable policy needs explicit overrides and likely per-environment calibration. They do not rescue a fixed global backend: no explicit backend dominates across layout, scale, and environment.

\subsection{External generalization}

The current evaluation uses a model ladder drawn from a single checkpoint family (Qwen3, supplemented by SmolLM3-3B). This is a deliberate design choice: by holding the architecture family roughly constant, scale and format become the dominant variables. However, the selector's thresholds were derived from and evaluated against this same set. A plausible reviewer concern is whether the policy generalizes to checkpoints with different shard geometries. Validation on held-out models from different families (e.g., Llama-3.1-8B, Mistral-7B-Instruct-v0.3) would strengthen the external validity claim. Even imperfect transfer would be a publishable insight: it would identify which workload features need refinement.

\subsection{Future work}

The next step is not to claim a perfect heuristic. It is to expand calibration: more local-NVMe hosts, filesystem and mount-option sweeps, direct-I/O variants, object-store backends, additional checkpoint formats, and learned or profile-guided selectors. Held-out model validation across different checkpoint families would test whether the workload-shape abstraction generalizes beyond the Qwen geometry. The current evidence justifies the abstraction (workload-shaped loading) and identifies where a stronger selector should improve.
