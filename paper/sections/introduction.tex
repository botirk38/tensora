\section{Introduction}
\label{sec:introduction}

An LLM serving request cannot produce a token until model weights have been made resident. For a 7B model in float16, the checkpoint payload is on the order of 14~GB; for larger models it quickly becomes tens of gigabytes. Serverless and multi-tenant deployments routinely evict idle models to recover GPU memory, so the next request may pay a cold load before inference can start. That load is user-visible and operationally expensive.

The common shorthand is to call this a storage-bandwidth problem. That shorthand hides the mechanism that matters to a loader. A checkpoint format defines the byte ranges that must be read. A backend defines how those ranges are submitted, batched, completed, and copied into buffers. A serving stack then adds Python object construction, tensor materialisation, CUDA initialisation, memory-pool setup, and scheduler overhead. A device with high peak throughput can still deliver poor load time if the request geometry and submission policy are mismatched.

This paper studies the following thesis:

\begin{quote}
\textbf{LLM checkpoint loading should be selected from workload shape.} Backend policy depends jointly on checkpoint layout, model scale, backend availability, and integration layer. A fixed global rule---always sync, always async, or always \texttt{io\_uring}---is the wrong abstraction.
\end{quote}

Two layouts expose why. SafeTensors checkpoints present mostly contiguous whole-file or multi-shard eager loads. ServerlessLLM-style checkpoints present partitioned range loads with a central index and many grouped reads. These are not small variations of one workload; they induce different I/O regimes. Scale then moves the crossover within a layout: a policy that is effective for a small single-shard load may leave throughput unused on a large multi-shard checkpoint.

We built Tensora\footnote{Tensora is this paper's checkpoint-loading framework (Rust crate \texttt{tensora}, Python package \texttt{tensora\_py}). It is unrelated to Google's TensorStore library for array storage.}, an open checkpoint-loading framework and evaluation harness, to measure this effect under controlled and integrated conditions. Tensora exposes synchronous POSIX I/O, Tokio-based async loading, Linux \texttt{io\_uring}, and a feature-driven \texttt{default} policy under common format abstractions. We evaluate six public checkpoints from 0.6B to 32B parameters, two layouts, and three layers: Rust backend attribution, Python bindings, and a custom vLLM integration. The primary environment is an H100 host with local NVMe and working \texttt{io\_uring}; H200 and A100 container environments with network-attached storage and blocked \texttt{io\_uring} test capability-gated fallback behaviour.

The data support a policy claim rather than a universal-loader claim. On H100 SafeTensors, explicit sync wins at the small end of the ladder, while explicit \texttt{io\_uring} wins from Qwen3-8B through Qwen3-32B. On H100 ServerlessLLM, sync is never the fastest backend; async, \texttt{io\_uring}, and default read planning dominate different rows. Python and vLLM change rankings because they measure different constructs, but loader choice still matters: in vLLM cold \texttt{load\_only} runs, Tensora's adaptive SafeTensors path reduces initialisation by 37--48\% relative to vLLM's native loader for the 8B and 32B checkpoints. Cross-environment results preserve the strongest structural fact---small SafeTensors models favour sync---while showing that the large-scale winner depends on kernel and storage context.

\textbf{Contributions.}
\begin{enumerate}
  \item A workload-shape characterisation of LLM checkpoint loading: format-level request geometry, not peak bandwidth alone, determines which backend policy is plausible.
  \item A controlled Rust-layer measurement matrix comparing sync, async, \texttt{io\_uring}, and adaptive default across SafeTensors and ServerlessLLM-style layouts.
  \item Cross-layer evidence showing how Python bindings and vLLM integration alter, but do not erase, backend effects.
  \item Cross-environment validation across H100, H200, and A100 systems showing that ordering regimes are more portable than the identity of the winning backend.
  \item A practical policy recommendation: use a format-aware adaptive selector with explicit overrides, capability checks, and measured workload features instead of a universal I/O backend.
\end{enumerate}
