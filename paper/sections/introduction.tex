\section{Introduction}
\label{sec:introduction}

An LLM serving request cannot begin until weights reach GPU memory. A 7B parameter model in float16 requires roughly 14~GB; serverless and multi-tenant environments evict idle models to reclaim memory, so every new request may trigger a cold load. This cold-start latency is directly visible to users, and its dominant component is I/O: reading checkpoint bytes from storage.

Peak NVMe bandwidth is only an upper bound. Checkpoint layout determines whether reads are contiguous or fragmented; the I/O backend determines how those reads are submitted, batched, and completed; Python bindings and serving engines add integration overhead that can mask backend-level differences. The gap between peak hardware and realized loading throughput is a systems problem in how reads are scheduled, not just how fast the storage device can transfer data.

\textbf{Thesis.} Backend policy for LLM checkpoint loading cannot be divorced from checkpoint layout or integration layer. A single fixed backend---always synchronous, always async, always io\_uring---is structurally suboptimal because two layouts produce fundamentally different I/O geometries: contiguous shard loads that benefit from bulk parallelism, and partitioned range loads dominated by fragmentation and grouping. Scale further shifts the optimum within each layout.

\textbf{Evidence.} Using Tensora\footnote{Tensora: this paper's checkpoint-loading framework (Rust crate \texttt{tensora}, Python package \texttt{tensora\_py}). It is unrelated to Google's TensorStore library for array storage.}, an open checkpoint-loading measurement framework, we evaluate four backend policies (synchronous POSIX I/O, Tokio-based async, io\_uring, and an adaptive default) across two formats (SafeTensors contiguous multi-shard and ServerlessLLM-style partitioned layouts) at three layers (Rust microbenchmark, Python bindings, vLLM integration). The SafeTensors Rust matrix exhibits a clear crossover: synchronous loading is fastest at small scales (0.6B--4B parameters), while io\_uring dominates from 8B parameters upward. For ServerlessLLM, synchronous loading is never the winner. Python tensor materialization adds enough overhead to reorder backends; vLLM initialization amplifies loader effects by 37--48\% relative to the native loader at 8B and 32B scales.

\textbf{Contributions.}
\begin{enumerate}
  \item A workload-shape characterization of LLM checkpoint loading showing that format determines I/O request geometry, which changes the optimal backend policy.
  \item A controlled measurement study comparing four backend policies across two checkpoint layouts, producing regime maps showing format-dependent crossovers.
  \item Cross-layer integration evidence from Python bindings and vLLM showing that backend rankings change across layers and that adaptive policy survives this reordering.
  \item An operational framework: deploy a format-aware adaptive policy with explicit overrides, grounded in measured workload features rather than API names.
\end{enumerate}
