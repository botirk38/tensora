\section{Introduction}
\label{sec:introduction}

An LLM serving request cannot generate a token until model weights are resident. In serverless and multi-tenant settings, idle models are commonly evicted to recover scarce accelerator memory; the next request then pays a cold load before inference begins. A 7B float16 checkpoint is already on the order of 14~GB, and the checkpoints in this paper reach beyond 65~GB of weight bytes. Cold loading is therefore not a startup footnote; it is part of the user-visible serving path.

The usual shorthand is to call this a bandwidth problem. That shorthand is incomplete. A loader does not read an abstract number of gigabytes; it executes a concrete read plan. The \emph{checkpoint format} decides whether bytes appear as a few contiguous shards or as many indexed ranges across partition files. The I/O backend decides whether those reads are submitted through blocking POSIX calls or an async runtime. Request geometry and submission method interact: the same backend can be fastest on one format and slowest on another.

Existing LLM serving systems and checkpoint libraries typically treat loading as a fixed initialization cost with a single code path. No prior work, to our knowledge, systematically evaluates I/O backend choice for LLM checkpoint loading as a function of checkpoint format and model scale. This gap matters because the optimal backend is not stable across the parameter space.

This paper evaluates I/O backends across two checkpoint layouts and six model scales on Modal H100 hardware. Our central finding is:

\begin{quote}
\textbf{Checkpoint format determines the optimal I/O backend.} SafeTensors (contiguous shards) and ServerlessLLM-style partitioning (indexed range reads) create different I/O problems, and the best backend shifts accordingly. A fixed rule such as ``always sync'' or ``always async'' leaves performance on the table.
\end{quote}

SafeTensors presents mostly whole-file or multi-shard eager loading: after metadata parsing, the loader must move large contiguous byte regions. ServerlessLLM-style partitioning presents a different workload: tensors are reconstructed from indexed partition files, creating many grouped range reads. These layouts are not cosmetic alternatives; they expose different I/O regimes. Scale then moves the boundary within a regime: a backend with low orchestration overhead can win for a small single-shard load, while a backend that sustains deeper submission queues can win for a large multi-shard load.

We built Tensora\footnote{Tensora is this paper's checkpoint-loading framework (Rust crate \texttt{tensora}, Python package \texttt{tensora\_py}). It is unrelated to Google's TensorStore library for array storage.}, an open framework for evaluating these regimes. Tensora exposes sync, async, and an adaptive default backend under shared format abstractions, enabling controlled backend attribution. We evaluate six public checkpoints from 0.6B to 32B parameters across two layouts, with five cold-cache repetitions per cell on Modal H100 containers. An external baseline (the upstream Hugging Face \texttt{safetensors} crate) and vLLM integration benchmarks provide end-to-end context. Held-out validation on Mistral-7B-Instruct tests generalisation beyond the primary model family.

On SafeTensors, async consistently outperforms sync across all scales. On ServerlessLLM, sync is never fastest; async leads at every scale. vLLM integration reorders absolute rankings but preserves the practical consequence: Tensora's async SafeTensors path reduces cold initialization relative to vLLM's native loader, with the advantage growing at larger model sizes.

\textbf{Contributions.}
\begin{enumerate}
  \item \textbf{Format-driven backend dependence.} We evaluate sync and async I/O backends across two checkpoint layouts and six model scales, showing that the best backend depends on format: async dominates SafeTensors at all scales, and sync never wins on ServerlessLLM. To our knowledge, this is the first controlled evidence that LLM checkpoint loading is format-dependent rather than bandwidth-bound.
  \item \textbf{Scale interaction.} We show that the performance gap between backends grows with model scale for both formats, and that an adaptive selector can match or approach the best explicit backend at each point in the format$\times$scale space.
  \item \textbf{External baseline and integration evidence.} We compare Tensora's backends against the upstream Hugging Face \texttt{safetensors} crate and vLLM's native loader, showing that backend-aware loading provides practical improvements that propagate through the serving stack.
\end{enumerate}
