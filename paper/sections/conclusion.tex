\section{Conclusion}
\label{sec:conclusion}

LLM checkpoint loading is not governed by storage bandwidth alone. It is governed by workload shape: the layout of bytes on disk, the scale of the checkpoint, the backend interfaces available on the host, and the overhead imposed by the integration layer.

This paper measured that shape with Tensora across two checkpoint layouts, four backend policies, three measurement layers, and three hardware environments. The evidence rejects a fixed global backend. SafeTensors shifts from sync-favoured small loads to \texttt{io\_uring}-favoured large H100 loads. ServerlessLLM-style partitioning changes the ranking so sync is never fastest in the H100 matrix. Python and vLLM reorder the backend story but preserve deployment relevance: loader choice affects public API time and serving initialisation. Cross-environment validation shows that regimes transfer better than exact winners.

The practical conclusion is therefore not ``use \texttt{io\_uring}'' or ``use sync.'' It is: load by design. Select a backend from measured workload features, gate it by runtime capability, and expose explicit overrides for boundary cases. A format-aware adaptive policy is the right abstraction for checkpoint loading because the workload itself changes the answer.
