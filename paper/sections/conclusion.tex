\section{Conclusion}
\label{sec:conclusion}

Checkpoint loading for LLM serving exhibits format-dependent performance regimes: the optimal I/O backend changes systematically with checkpoint layout and model scale. This paper presents a controlled evaluation of sync and async I/O backends across two checkpoint layouts and six model scales on Modal H100 hardware, backed by five cold-cache repetitions per cell.

SafeTensors loading consistently favours async over sync, with the advantage growing at larger model scales. ServerlessLLM-style partitioning amplifies this effect: sync never wins when the read geometry is range-dominated. An adaptive backend selector matches or approaches the best explicit backend at each point in the format$\times$scale space. The upstream Hugging Face \texttt{safetensors} crate baseline confirms that Tensora's backend-aware loading provides practical advantage over the default loading path. vLLM integration benchmarks show that backend choice propagates to serving initialization: Tensora's async SafeTensors path reduces cold \texttt{load\_only} time relative to vLLM's native loader.

These findings suggest that checkpoint loading should be treated as a format-dependent I/O evaluation problem rather than a fixed initialization cost. A serving system that exposes backend selection matched to checkpoint layout and model scale can reduce cold-start latency without changes to the model, the checkpoint format, or the serving scheduler.
