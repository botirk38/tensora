\section{Conclusion}
\label{sec:conclusion}

LLM checkpoint loading is not a storage-bandwidth problem---it is a workload-shape problem. The I/O request geometry determined by checkpoint layout changes which backend policy is optimal, and integration layers (Python, vLLM) can reorder but not eliminate the ranking.

This paper presented a measurement-driven study of four backend policies across two checkpoint layouts and three integration layers. Three findings are robust. First, SafeTensors exhibits a crossover from sync-favored small loads to io\_uring-favored large multi-shard loads; no fixed backend tracks both regimes. Second, the partitioned ServerlessLLM layout reverses the ranking: sync never wins, and the trade-off is between async and io\_uring depending on partition structure. Third, an adaptive default policy captures the dominant regime in 9 of 12 Rust-layer cells; its known misses on medium-scale rows identify precisely where explicit overrides add value. Python and vLLM integration confirms that adaptive policy outperforms any single fixed backend in deployment.

The contribution is not a single optimal loader. It is evidence that backend choice depends jointly on layout, scale, and integration layer, and that a format-aware adaptive policy with explicit overrides provides practical value by automating the dominant patterns. Deploy workload-shaped loading, not a universal default.
