\section{Tensora Design}
\label{sec:design}

This section describes \emph{Tensora}, the experimental framework used for the measurement study. The framework is intentionally narrow: it makes checkpoint-loading workload shape observable, runs multiple backends over comparable read plans, and carries the same policy into Python and vLLM integration tests.

\subsection{Design Goals}

Tensora was designed around four requirements.
\begin{enumerate}
  \item \textbf{Comparable workloads.} Multiple formats and backends must share a common loading abstraction so backend changes do not silently change the logical work.
  \item \textbf{Separation from measurement.} Cold-cache rituals, timing, and result export belong to the harness, not to the loader.
  \item \textbf{Explicit concurrency.} Queue depth, worker fanout, chunking, and range grouping are first-class design variables.
  \item \textbf{Cross-layer reach.} The same core must be callable from Rust benchmarks, Python bindings, and a serving-engine integration.
\end{enumerate}

These requirements led to a layered architecture that separates format parsing, policy selection, I/O execution, and evaluation.

\subsection{System Architecture}

Tensora has three layers. The \textbf{format layer} parses checkpoint metadata and produces a request-local view of the workload: which files and byte ranges must be read to materialise tensors. The \textbf{policy/backend layer} chooses a backend and executes that read plan using backend-specific planning such as file chunking, range coalescing, and in-flight depth control. The \textbf{evaluation layer} contains the Rust profiler, Python benchmarks, and vLLM integration benchmarks. It observes the boundary; it is not on the hot path.

ServerlessLLM follows the same pattern after compiling an index-derived partition plan.

\subsection{Common Backend Interface}

All eager-loading backends implement the same conceptual surface:

\begin{itemize}
\item \texttt{load(path)} for a full file,
\item \texttt{load\_batch(paths)} for multiple whole-file loads,
\item \texttt{load\_range(path, offset, len)} for one byte range, and
\item \texttt{load\_range\_batch(requests)} for many ranges.
\end{itemize}

This interface is deliberately narrow. It covers whole-file SafeTensors loading and range-oriented ServerlessLLM loading without exposing format-specific shortcuts to one backend. The benchmark comparison therefore remains about submission and planning strategy.

\subsection{Workload Features}

The format layer extracts workload features used both for analysis and for the adaptive policy:

\begin{itemize}
\item \textbf{Total bytes}: aggregate checkpoint size across shards or partitions.
\item \textbf{Shard / partition count}: number of discrete files or partition files.
\item \textbf{Request count}: number of byte-range reads needed to materialise tensors.
\item \textbf{Contiguity}: whether the load is dominated by whole-file sequential reads or grouped ranges.
\item \textbf{Format family}: SafeTensors or ServerlessLLM-style partitioning.
\end{itemize}

SafeTensors checkpoints are characterised by low request count and high contiguity: tensors live in whole files or contiguous shards. ServerlessLLM-style checkpoints have higher partition and range structure: tensors are reconstructed from indexed partition files. These differences are the root cause of the backend-ordering reversals observed in Section~\ref{sec:results}.

\subsection{Adaptive Loading Policy}

The central design decision is that \texttt{default} is an adaptive heuristic, not an alias for one backend. The Rust matrix (Tables~\ref{tab:rust-safetensors} and~\ref{tab:rust-serverlessllm}) shows why: no single explicit backend is best across both layouts and all scales.

The \texttt{default} policy operates in three stages. First, the format layer computes workload statistics: total bytes, shard or partition count, and coarse contiguity. Second, \texttt{choose\_load\_backend(stats)} maps those features to a backend while consulting \texttt{backend\_capabilities()} so unavailable interfaces, such as \texttt{io\_uring} under container security policy, are excluded. Third, backend-specific planning applies chunking, coalescing, and in-flight limits before execution.

This means \texttt{default} is not merely a selector among explicit columns. It combines backend choice with execution-time planning. That is intentional: a production loader should both choose a suitable backend and execute it with an appropriate plan. The oracle analysis in Section~\ref{sec:oracle-analysis} reports where this integrated default beats explicit paths and where it misses.

For \texttt{SafeTensors}, the heuristic maps shard count and total bytes to a regime decision: synchronous execution for smaller eager loads, and \texttt{io\_uring} for larger multi-shard loads when available. If \texttt{io\_uring} is blocked, large candidates fall back to async. For \texttt{ServerlessLLM}, the feature set is partition and request structure; the heuristic does not route to synchronous I/O because the Rust matrix shows sync never wins on this layout.

The selector is feature-driven, not a universal byte threshold. Exact constants are implementation details tied to the measured revision; the paper claims regimes, not magic numbers. Figure~\ref{fig:policy-sketch} summarises control flow. Table~\ref{tab:oracle} quantifies boundedness: \texttt{default} is strong in stable regimes and weakest in medium-scale rows where additional signals would be needed.

\subsection{Backend Designs}

The synchronous backend uses blocking POSIX reads. It is not a deliberately weak serial baseline: large files can be split into chunks and loaded with host-side thread parallelism. This gives sync a practical implementation and makes its small-model wins meaningful.

The async backend is built on Tokio. Its design is strongest for grouped and range-heavy workloads rather than large whole-file eager loads. It groups work per file, uses dynamic batching, and limits unnecessary task fanout. This explains why async is weak for H100 SafeTensors but often strong for ServerlessLLM and for \texttt{io\_uring}-restricted environments.

The \texttt{io\_uring} backend uses persistent rings, batched submissions, lazy initialisation, dynamic ring sizing, and workload-driven planning for chunk size and in-flight depth. Its benefit depends on sustaining enough queued work; the paper does not claim that the API name alone guarantees speed.

The mmap-backed path is retained as a useful lazy-access mode, especially for Python and vLLM experiments where a caller may not need to materialise every tensor immediately. It is not a main eager-loading solution to cold start: mmap optimises selective access, while this paper studies aggressive eager loading.

\subsection{Checkpoint Formats}

Tensora supports two checkpoint layouts. \texttt{SafeTensors} stores tensors in a whole-file or multi-shard layout. Once the header is parsed, the loader sees a largely contiguous eager-loading workload. The \texttt{ServerlessLLM} layout distributes tensors across partition files with a central index, creating a fragmented range-oriented workload. Supporting both formats exposes two I/O regimes under one framework.

\subsection{Python Bindings and vLLM Integration}

Tensora exposes Python bindings implemented with PyO3. The public API intentionally stays small: a \texttt{default} eager path, explicit \texttt{sync} and \texttt{async} eager paths, and lazy \texttt{open\_*} mmap-backed handles. On Linux, \texttt{io\_uring} remains behind \texttt{default} in the public Python API; benchmark-only explicit variants expose additional backend behaviour for the vLLM layer.

One binding decision was especially important: the async Python path uses a shared Tokio runtime rather than constructing a fresh runtime per call. This reduced binding overhead and improved the public default path in Table~\ref{tab:python-optimisation}.

The vLLM integration uses a custom model-loader path. vLLM is a useful evaluation target because it separates model initialisation, time-to-first-token, and steady-state decode. The key constraint is that vLLM's loader interface is synchronous, so the benchmark integration follows that interface rather than inventing an async serving path that the engine does not expose.

Cold-cache control is implemented in the experiment harness, not inside the loader. The Linux shell runs \texttt{sync} and \texttt{drop\_caches} before benchmark invocations. This keeps the loader responsible for loading checkpoints and the harness responsible for creating cold and warm conditions.
