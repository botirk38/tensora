\section{Tensora Design}
\label{sec:design}

Tensora is the measurement vehicle for this paper. Its design goal is not to hide loading behind a single convenient API, but to make the workload shape, backend choice, and integration overhead explicit enough to measure.

\subsection{Design principles}

Tensora follows four principles.
\begin{enumerate}
  \item \textbf{One logical workload, many executions.} A format parser should produce the same tensor read plan regardless of backend, so timing differences reflect submission and planning policy rather than different work.
  \item \textbf{Backend policy is explicit.} Sync, async, \texttt{io\_uring}, and \texttt{default} are separate labels in the harness. The adaptive path is not allowed to disappear into an opaque fast path.
  \item \textbf{Measurement stays outside the loader.} Cache drops, timing loops, environment capture, and export formatting are harness responsibilities. Loader code should load checkpoints, not manufacture benchmark conditions.
  \item \textbf{The policy must cross layers.} A selector that only works in a Rust microbenchmark is incomplete. Tensora carries the same core into Python bindings and a vLLM loader integration.
\end{enumerate}

\subsection{Architecture}

Tensora separates format parsing, backend execution, policy selection, and evaluation (Figure~\ref{fig:tensora-arch}). The \textbf{format layer} parses checkpoint metadata and produces a request-local read plan: files, byte ranges, tensor names, and expected sizes. The \textbf{policy/backend layer} either executes an explicit backend selected by the caller or invokes the adaptive selector. The \textbf{evaluation layer} contains the Rust profiler, Python benchmarks, and vLLM experiments; it observes boundaries without being part of the production load path.

At the SafeTensors layer, the default load path is \texttt{Model::load(path)} $\rightarrow$ workload statistics $\rightarrow$ \texttt{choose\_load\_backend(stats)} $\rightarrow$ \texttt{load\_sync}, \texttt{load\_async}, or \texttt{load\_io\_uring}. The Linux implementation checks \texttt{backend\_capabilities()} before selecting \texttt{io\_uring}. ServerlessLLM follows the same structure after \texttt{compile\_load\_plan(directory)} builds an index-derived partition plan.

\subsection{Common backend surface}

All eager backends implement the same conceptual operations:

\begin{itemize}
\item \texttt{load(path)} for one full file,
\item \texttt{load\_batch(paths)} for multiple whole-file loads,
\item \texttt{load\_range(path, offset, len)} for one byte range, and
\item \texttt{load\_range\_batch(requests)} for grouped range reads.
\end{itemize}

This surface is deliberately small. SafeTensors mostly exercises whole-file and batch loading; ServerlessLLM exercises range batches. Backend implementations may chunk, group, coalesce, or tune queue depth internally, but they must satisfy the same logical tensor plan.

\subsection{Workload features}

The selector and analysis use features available before reading tensor payloads:

\begin{itemize}
\item \textbf{Total bytes}: aggregate checkpoint payload size.
\item \textbf{Shard or partition count}: number of files contributing to the load.
\item \textbf{Average shard or partition size}: a coarse scale signal for fanout and queue-depth potential.
\item \textbf{Request structure}: whole-file sequential reads versus indexed ranges.
\item \textbf{Backend capability}: whether \texttt{io\_uring} can be initialized on the host.
\end{itemize}

These features intentionally stop short of learning a per-file oracle. The policy should be explainable to an operator: small contiguous loads tend to favor low-overhead sync; large multi-shard loads can benefit from deeper submission; partitioned range loads need grouping and batching.

\subsection{Adaptive loading policy}

The \texttt{default} path is an adaptive policy plus read planning, not a synonym for one explicit backend. It first extracts workload features, then applies a capability-gated selector, then lets the chosen backend plan execution. On Linux, \texttt{choose\_load\_backend\_with\_capabilities(stats, capabilities)} prevents \texttt{io\_uring} selection when ring creation is blocked or unsupported. This matters in the H200 and A100 environments, where container policy returns \texttt{EPERM}.

For SafeTensors, the selector uses total bytes, shard count, and average shard size to separate small contiguous loads from large multi-shard loads. For ServerlessLLM, it uses partition and range structure; the H100 matrix shows that a sync-first rule from SafeTensors should not be imported into this layout. The policy is therefore format-aware by construction.

The selector is intentionally not presented as an oracle. Section~\ref{sec:oracle-analysis} reports where it wins, where it is near parity, and where it misses. Those misses motivate explicit overrides. A deployable adaptive loader should expose both: a good default for stable regimes and manual labels for known boundary cases.

\subsection{Backend implementations}

The \textbf{sync} backend uses blocking POSIX reads with practical host-side parallelism for large files. It is not a strawman serial baseline; its small-model wins are meaningful because it has low orchestration overhead.

The \textbf{async} backend is built on Tokio. It groups work per file, batches dynamically, and avoids unnecessary task fanout. It is weak for H100 SafeTensors, where whole-file eager loads do not justify async overhead, but strong for partitioned and network-storage cases.

The \textbf{\texttt{io\_uring}} backend uses persistent rings, batched submissions, lazy initialization, dynamic ring sizing, and workload-dependent chunk and depth planning. Its value depends on enough queued work; the paper never claims that \texttt{io\_uring} is intrinsically faster by API name alone.

The mmap-backed path remains useful for lazy selective access, especially through Python handles, but it is not the main eager cold-start mechanism evaluated in the Rust matrices. Eager loading and mmap answer different questions.

\subsection{Formats and integrations}

Tensora supports two layouts because they expose different regimes. \texttt{SafeTensors} presents whole-file or multi-shard contiguous loading. \texttt{ServerlessLLM} presents an indexed partition layout with range-oriented reads. Supporting both under one framework is what makes the layout-dependent policy claim measurable.

Python bindings are implemented with PyO3 and expose a small API: adaptive eager load, explicit sync and async eager load, and lazy \texttt{open\_*} handles. The async Python path uses a shared Tokio runtime rather than constructing one per call. The vLLM integration uses a custom loader path and follows vLLM's synchronous loader boundary, so the serving benchmark reflects the interface available to real integrations rather than an invented async serving path.
