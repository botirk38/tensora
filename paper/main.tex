\begin{abstract}
Cold-start latency in large language model (LLM) serving is usually treated as a bandwidth problem, but the optimal I/O strategy depends on \emph{checkpoint format}. A contiguous SafeTensors layout creates a different I/O problem than a partitioned ServerlessLLM-style layout; the best userspace backend shifts accordingly.

We present a systematic evaluation using Tensora, an open checkpoint-loading framework designed for backend attribution. On Modal H100 hardware we compare sync (POSIX) and async (Tokio) backends across two layouts and six public checkpoints from 0.6\,B to 32\,B parameters, with five cold-cache repetitions per cell. An external baseline (Hugging Face \texttt{safetensors} crate) and vLLM integration benchmarks provide end-to-end context. Held-out validation on Mistral-7B-Instruct confirms that regime structure generalises beyond the primary model family.

Four findings emerge. First, SafeTensors loading exhibits a consistent pattern: async outperforms sync across all scales, with the advantage growing from 1.13$\times$ at 0.6\,B to 2.40$\times$ at 32\,B. Second, ServerlessLLM-style partitioning strongly favours async at every scale; sync never wins when the read geometry is range-dominated. Third, Tensora's async backend is 1.6--13.8$\times$ faster than the stock Hugging Face SafeTensors loader, confirming practical improvement over Python-level loading. Fourth, vLLM end-to-end measurements show that raw loading gains are partially absorbed by engine overhead (graph construction, KV-cache allocation, autotuning), with the advantage most visible at small model scales where loading dominates initialization. These findings show that checkpoint format, not just bandwidth, should drive I/O backend selection in serving systems.
\end{abstract}
