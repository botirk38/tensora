\begin{abstract}
Cold-start latency in large language model (LLM) serving is often treated as a storage-throughput problem: buy faster NVMe and load checkpoints faster. This paper shows that this view is incomplete. Checkpoint layout determines the I/O request geometry exposed to the operating system, backend policy determines how those requests are submitted, and integration layers such as Python and vLLM can amplify or hide backend-level effects. The right loader is therefore workload-shaped, not universal.

We present Tensora, an open measurement framework for checkpoint loading, and use it to evaluate synchronous POSIX I/O, Tokio-based async I/O, Linux \texttt{io\_uring}, and an adaptive default policy. The study spans two layouts---contiguous SafeTensors shards and a partitioned ServerlessLLM-style layout---and three measurement layers: Rust backend attribution, Python bindings, and a custom vLLM integration. The Rust matrix covers six public checkpoints from 0.6B to 32B parameters. Cross-environment validation on H100, H200, and A100 systems tests whether ordering regimes survive changes in hardware, storage topology, and kernel policy.

Four findings emerge. First, SafeTensors has a scale-dependent crossover: synchronous loading is fastest at small scales, while \texttt{io\_uring} wins on the H100 host for 8B--32B multi-shard checkpoints. Second, the partitioned ServerlessLLM layout changes the problem: synchronous loading is never the fastest backend in the H100 matrix, and the relevant trade-off shifts to async, \texttt{io\_uring}, and default read planning. Third, integration changes rankings but does not eliminate loader impact. Python tensor materialisation adds substantial overhead, and in vLLM cold \texttt{load\_only} runs Tensora's adaptive SafeTensors path reduces initialisation time by 37--48\% relative to the native loader for 8B and 32B models. Fourth, regime structure is more portable than backend identity: small SafeTensors models favour sync across all three environments, but the mid-to-large-scale winner depends on whether \texttt{io\_uring} is available and on storage topology.

The contribution is an evidence-backed policy result: deploy a format-aware adaptive loader with explicit overrides, selected from measured features such as bytes, shard count, partition count, backend availability, and integration layer. Do not choose an I/O API name as a universal default.
\end{abstract}
