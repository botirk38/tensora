\begin{abstract}
LLM cold-start latency is dominated by reading checkpoint weights from storage into GPU memory. Peak NVMe bandwidth is only an upper bound: realized throughput depends on the I/O request geometry determined by checkpoint layout, the submission policy of the loading backend, and integration overhead from the serving stack. This paper shows that checkpoint layout determines request geometry, which changes the optimal loading backend across scales and integration layers.

Using Tensora, an open checkpoint-loading measurement framework, we evaluate synchronous POSIX I/O, Tokio-based async, io\_uring, and an adaptive default policy across two layouts (SafeTensors contiguous shards and ServerlessLLM-style partitioned files) at three layers: a Rust microbenchmark for pure backend attribution (six public checkpoints from 0.6B to 32B parameters), Python bindings for user-facing overhead, and a custom vLLM integration for end-to-end serving impact. Cross-environment validation on H200 and A100 containers---where kernel security policy blocks \texttt{io\_uring}---tests whether findings transfer.

Four findings emerge. First, SafeTensors exhibits a clear crossover: synchronous loading is fastest for small models, while io\_uring dominates for multi-shard checkpoints at 8B parameters and above; no fixed backend tracks both regimes. Second, the partitioned ServerlessLLM layout reverses the ranking---synchronous loading never wins, and the relevant trade-off is between async and io\_uring depending on partition structure. Third, Python tensor materialization and vLLM initialization overhead change which backend appears fastest, but adaptive policy consistently outperforms any single fixed choice. In vLLM cold start, Tensora's adaptive SafeTensors path reduces initialization time by 37--48\% compared to the native loader for 8B and 32B models. Fourth, the regime structure (small models favour sync, large models favour another backend) persists across all three environments, but the identity of the large-scale winner depends on kernel interface availability and storage topology---making the adaptive selector more valuable than any fixed policy.

The operational conclusion is not a single universal loader. It is a format-aware adaptive policy with explicit overrides, grounded in measured workload features---bytes, shard count, partition count, and integration layer---rather than API names alone.
\end{abstract}
