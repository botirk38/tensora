\begin{abstract}
Cold-start latency in large language model (LLM) serving is often reduced to a hardware question: how quickly can storage stream checkpoint bytes? This paper argues that the better abstraction is \emph{workload shape}. A checkpoint layout determines the file and range requests exposed to the operating system; an I/O backend determines how those requests are submitted and completed; Python bindings and serving engines determine how much of any backend gain reaches the user. Consequently, the best loader is not a universal API choice, but a policy conditioned on format, scale, runtime capability, and integration layer.

We present Tensora, an open checkpoint-loading framework and measurement harness for evaluating that policy space. Tensora runs synchronous POSIX I/O, Tokio-based async I/O, Linux \texttt{io\_uring}, and an adaptive default policy over two layouts: contiguous SafeTensors shards and a partitioned ServerlessLLM-style layout. The evaluation spans three layers---Rust backend attribution, Python bindings, and a custom vLLM integration---six public checkpoints from 0.6B to 32B parameters, and three environments: an H100 host with local NVMe and working \texttt{io\_uring}, plus H200 and A100 container environments where \texttt{io\_uring} is blocked.

The results show four regime changes. First, H100 SafeTensors has a scale crossover: sync is fastest at small scales, while \texttt{io\_uring} is fastest for 8B--32B multi-shard checkpoints. Second, the ServerlessLLM-style layout changes the ranking: sync is never fastest in the H100 matrix, and async, \texttt{io\_uring}, and default read planning trade places depending on partition structure. Third, integration layers reorder but preserve loader impact: Python materialization dominates absolute time, while vLLM cold \texttt{load\_only} improves by 37--48\% for 8B and 32B models with Tensora's adaptive SafeTensors path relative to vLLM's native loader. Fourth, ordering regimes are more portable than backend identities: small SafeTensors loads favor sync across all three environments, but the mid-to-large-scale winner changes with storage topology and backend availability.

The operational conclusion is deliberately bounded: use a format-aware adaptive loader with explicit overrides and capability checks. Do not choose sync, async, or \texttt{io\_uring} as a universal default by name alone.
\end{abstract}
