Cold-start latency in large language model serving is often treated as a storage-throughput problem. This paper shows that view is incomplete: checkpoint layout determines I/O request geometry, backend policy determines how requests are submitted, and integration layers such as Python and vLLM can amplify or hide backend-level effects.

We present Tensora, an open checkpoint-loading framework and evaluation harness for Linux. Tensora supports SafeTensors and a partitioned ServerlessLLM-style layout, with eager backends spanning synchronous POSIX I/O, Tokio-based async loading, Linux io_uring, and an adaptive default policy. Experiments cover six public checkpoints from 0.6B to 32B parameters, Rust backend-attribution benchmarks, Python bindings, vLLM integration, and cross-environment validation on H100, H200, and A100 systems.

Findings. For contiguous SafeTensors, explicit synchronous loading wins at small scales, while explicit io_uring wins on the H100 host for 8B--32B multi-shard checkpoints. For partitioned ServerlessLLM, synchronous loading is never fastest in the H100 matrix; the trade-off shifts to async, io_uring, and default read planning. Python tensor materialisation and vLLM initialisation reorder backend rankings but do not eliminate loader impact; in vLLM cold load_only runs, Tensora's adaptive SafeTensors path reduces initialisation time by 37--48% relative to the native loader for 8B and 32B models. Cross-environment results show that regime structure is more portable than the identity of the winning backend. The operational recommendation is therefore a format-aware adaptive loader with explicit overrides, not one universal I/O API.
