for preceding element\n\nWith the proliferation of large language model (LLM) variants,\ndevelopers are turning to serverless computing for cost-efficient LLM deployment.\nHowever, public cloud providers often struggle to provide performance guarantees for serverless LLM serving due to significant cold start latency caused by substantial model sizes and complex runtime dependencies.\nTo address this problem, we present HydraServe,\na serverless LLM serving system designed to minimize cold start latency in public clouds.\nHydraServe proactively distributes models across servers to quickly fetch them,\nand overlaps cold-start stages within workers to reduce startup latency.\nAdditionally, HydraServe strategically places workers across GPUs to avoid network contention among cold-start instances.\nTo minimize resource consumption during cold starts,\nHydraServe further introduces pipeline consolidation that can merge groups of workers into individual serving endpoints.\nOur comprehensive evaluations under diverse settings demonstrate that HydraServe reduces the cold start latency by 1.7×\\\\times– 4.7×\\\\times and improves service level objective [33](https://arxiv.org/html/2502.15524v2#bib.bib33 \"\")\\].\nIn serverless inference, the cloud platform automatically provisions GPUs for models based on the actual request load,\nand model owners are only charged for the resource consumption during request processing.\nThis approach offers significant cost savings for long-tail models with infrequent usage.\n\nReport issue for preceding element\n\nHowever, for cloud providers, delivering performance guarantees for serverless LLM users is challenging, primarily due to the long latency of cold starts, i.e., provisioning new workers to meet the current load.\nProduction traces reveal that bursty traffic patterns are prevalent in both LLM serving workloads \\[ [34](https://arxiv.org/html/2502.15524v2#bib.bib34 \"\")\\] and serverless workloads \\[ [35](https://arxiv.org/html/2502.15524v2#bib.bib35 \"\")\\], making cold starts inevitable in serverless LLM serving \\[ [30](https://arxiv.org/html/2502.15524v2#bib.bib30 \"\"), [32](https://arxiv.org/html/2502.15524v2#bib.bib32 \"\"), [31](https://arxiv.org/html/2502.15524v2#bib.bib31 \"\"), [33](https://arxiv.org/html/2502.15524v2#bib.bib33 \"\")\\].\nCold starts can directly impact service level objectives (SLOs) for LLM inference.\nFor instance, in our production environment, a cold-start instance produces the first token after more than 40 seconds, whereas subsequent generation takes only ∼\\\\sim30ms per token.\nThis contrast stresses the need for faster cold starts.\n\nReport issue for preceding element\n\nAddressing this problem is non-trivial, as LLM cold starts introduce two unique challenges compared to traditional serverless workloads.\nFirst, LLM applications require massive external state,\nwith model weights fetched on demand from remote registries during cold starts.\nSince serverless instances typically operate with constrained network bandwidth,\nmodel fetching becomes the primary bottleneck in cold start latency.\nSecond, the LLM runtime is highly complex, involving multiple specialized libraries including CUDA runtime \\[ [36](https://arxiv.org/html/2502.15524v2#bib.bib36 \"\")\\], AI frameworks \\[ [37](https://arxiv.org/html/2502.15524v2#bib.bib37 \"\"), [38](https://arxiv.org/html/2502.15524v2#bib.bib38 \"\")\\], and inference engines \\[ [39](https://arxiv.org/html/2502.15524v2#bib.bib39 \"\"), [40](https://arxiv.org/html/2502.15524v2#bib.bib40 \"\"), [41](https://arxiv.org/html/2502.15524v2#bib.bib41 \"\")\\].\nThe complex dependencies of these libraries lead to substantial container creation and library loading overhead \\[ [42](https://arxiv.org/html/2502.15524v2#bib.bib42 \"\")\\].\nIn public clouds, applications run in isolated containers with varying library versions, making it impractical to prepare runtimes in advance for every user.\n\nReport issue for preceding element\n\nPrior works focus on reducing the model fetching latency and propose two approaches: (1) caching LLMs in memory or SSDs \\[ [30](https://arxiv.org/html/2502.15524v2#bib.bib30 \"\"), [33](https://arxiv.org/html/2502.15524v2#bib.bib33 \"\"), [43](https://arxiv.org/html/2502.15524v2#bib.bib43 \"\")\\] and (2) retrieving model weights from active workers holding the same model \\[ [31](https://arxiv.org/html/2502.15524v2#bib.bib31 \"\"), [32](https://arxiv.org/html/2502.15524v2#bib.bib32 \"\"), [33](https://arxiv.org/html/2502.15524v2#bib.bib33 \"\")\\].\nWhile effective for frequently accessed models, these optimizations fall short for long-tail customers,\nwho benefit the most from the serverless paradigm \\[ [44](https://arxiv.org/html/2502.15524v2#bib.bib44 \"\")\\].\nAdditionally, retrieving data from peer workers requires high-bandwidth networking,\nwhich is not cost-efficient for serverless inference clusters.\n\nReport issue for preceding element\n\nTo this end, we present HydraServe, a serverless LLM serving system designed to minimize cold start latency in public clouds.\nHydraServe incorporates two key insights to mitigate the overhead of model fetching and runtime preparation.\nFirst, while a single server has limited bandwidth for model fetching, distributing workers across multiple servers enables efficient bandwidth aggregation.\nSecond, since model fetching and runtime preparation exhibit weak execution dependencies,\nthese stages can be parallelized rather than executed sequentially, thereby overlapping their latencies.\n\nReport issue for preceding element\n\nSpecifically, to distribute models across servers, we take advantage of the layered structure of LLMs.\nHydraServe partitions LLM layers across servers and exchanges intermediate results during inference.\nThis design, known as _pipeline parallelism_, has been well studied in distributed model training and inference \\[ [45](https://arxiv.org/html/2502.15524v2#bib.bib45 \"\"), [46](https://arxiv.org/html/2502.15524v2#bib.bib46 \"\"), [47](https://arxiv.org/html/2502.15524v2#bib.bib47 \"\"), [48](https://arxiv.org/html/2502.15524v2#bib.bib48 \"\"), [49](https://arxiv.org/html/2502.15524v2#bib.bib49 \"\"), [50](https://arxiv.org/html/2502.15524v2#bib.bib50 \"\"), [51](https://arxiv.org/html/2502.15524v2#bib.bib51 \"\"), [52](https://arxiv.org/html/2502.15524v2#bib.bib52 \"\"), [53](https://arxiv.org/html/2502.15524v2#bib.bib53 \"\"), [54](https://arxiv.org/html/2502.15524v2#bib.bib54 \"\")\\].\nPipeline parallelism is also used to quickly start inference when scaling up multiple workers \\[ [32](https://arxiv.org/html/2502.15524v2#bib.bib32 \"\")\\].\nHydraServe further adopts pipeline parallelism to reduce the blocking effect of model fetching—we proactively create a group of workers in parallel even if the model only needs one.\n\nReport issue for preceding element\n\nAlthough pipeline parallelism can effectively reduce the cold start latency for the initial inference,\nit may impact performance in subsequent rounds due to GPU sharing.\nTo address this, we propose _pipeline consolidation_.\nSpecifically, we allow workers to fetch and load the layers previously assigned to other workers in the background, eventually transitioning back to local inference with a fully-loaded model.\nDepending on the load, each worker can independently decide whether to release its resources or load the full model after cold starts.\nTherefore, with a single cold start, HydraServe can choose to only create a single worker in the end or scale up as many workers as the pipeline size, allowing for high elasticity.\n\nReport issue for preceding element\n\nHydraServe takes a three-level _hierarchical_ approach to realize the above optimizations.\nAt the _cluster level_, we design an algorithm to allocate resources for cold-start models based on the model size, cluster resource usage, and user SLOs.\nA network-contention-aware worker placement policy is adopted to avoid potential network contention.\nAt the _worker level_, we overlap remote-to-host model fetching, host-to-GPU model loading, and the initialization of container and GPU runtime to further reduce the cold start latency.\nFinally, at the _inference level_, we consolidate workers that are previously created in pipeline groups and migrate their runtime states to ensure peak performance with all weights loaded.\n\nReport issue for preceding element\n\nIn summary, we make the following contributions:\n\nReport issue for preceding element\n\n- •\n\n\nWe articulate the main bottleneck of LLM cold starts in public clouds and identify opportunities to mitigate the overhead.\n\nReport issue for preceding element\n\n- •\n\n\nWe propose a hierarchical approach that combines cluster-level, worker-level, and inference-level optimizations to reduce cold start latency for serverless LLM serving in public clouds.\n\nReport issue for preceding element\n\n- •\n\n\nWe implement HydraServe in both testbed and production environment. Experiments on real-world datasets show that HydraServe reduces the cold start latency by 1.7×\\\\times– 4.7×\\\\times and improves SLO attainment by 1.43×\\\\times–1.74×\\\\times compared to baselines.\nThe brownfield evaluation further confirms these gains, yielding an average latency reduction of 2.6×\\\\times.\n\nReport issue for preceding element\n\n\n## 2 Background and Motivation\n\nReport issue for preceding element\n\nThis section provides an overview of large language models (LLMs) and serverless LLM serving.\nWe highlight the critical issue of limited network bandwidth for LLM serving in public clouds,\nwhich can substantially increase cold start latency.\nWe then identify opportunities for mitigating this performance bottleneck in a cost-effective manner.\n\nReport issue for preceding element\n\n### 2.1 LLM Inference\n\nReport issue for preceding element\n\nAn LLM takes as input a sequence of tokens, called _prompt_, and generates an output sequence in an _autoregressive_ manner,\nproducing one token at a time based on both the prompt and previously generated outputs.\nThe core component of an LLM is the self-attention layer, which needs to compute the key, value, and query vectors of tokens in the input sequence.\nSince the key and value vectors of previous tokens remain unchanged during iterations,\nLLM serving systems usually cache these vectors to avoid redundant computation, known as _KV cache_.\nThe stage that generates the first token is called _prefill_, during which the key and value vectors of all tokens in the prompt are calculated and stored in GPU memory.\nSubsequent tokens are generated in the _decoding_ phase, which reuses the key-value cache and generates one token at a time.\n\nReport issue for preceding element\n\nIn LLM serving, users often specify service level objectives (SLOs) that represent performance expectations \\[ [55](https://arxiv.org/html/2502.15524v2#bib.bib55 \"\"), [56](https://arxiv.org/html/2502.15524v2#bib.bib56 \"\"), [57](https://arxiv.org/html/2502.15524v2#bib.bib57 \"\"), [51](https://arxiv.org/html/2502.15524v2#bib.bib51 \"\")\\].\nThese SLOs typically focus on two metrics: time to first token (TTFT) and time per output token (TPOT).\nTTFT measures the latency from request submission to the generation of the _first_ token,\nwhile TPOT represents the average time taken to generate each _subsequent_ token.\nThe relative importance of these metrics varies across different applications.\nFor example, real-time chatbots emphasize low TTFT, whereas article writing prefers lower TPOT \\[ [55](https://arxiv.org/html/2502.15524v2#bib.bib55 \"\")\\].\n\nReport issue for preceding element\n\nPipeline parallelism is a model parallelism strategy that is commonly used in LLM training to scale up involved GPUs \\[ [45](https://arxiv.org/html/2502.15524v2#bib.bib45 \"\"), [46](https://arxiv.org/html/2502.15524v2#bib.bib46 \"\"), [47](https://arxiv.org/html/2502.15524v2#bib.bib47 \"\"), [48](https://arxiv.org/html/2502.15524v2#bib.bib48 \"\"), [49](https://arxiv.org/html/2502.15524v2#bib.bib49 \"\"), [50](https://arxiv.org/html/2502.15524v2#bib.bib50 \"\"), [58](https://arxiv.org/html/2502.15524v2#bib.bib58 \"\")\\].\nThis approach distributes a model’s layers across multiple workers,\nwith intermediate results transmitted sequentially between workers during computation.\n\nReport issue for preceding element\n\n### 2.2 Serverless LLM Serving in Public Clouds\n\nReport issue for preceding on the GPU capabilities.\nSince serverless customers prioritize cost savings,\ncloud providers aim to minimize cost per GPU when configuring their infrastructure.\nThis cost-optimization strategy favors servers with reduced CPU, memory, and network resources.\n\nReport issue for preceding element\n\nTable [1](https://arxiv.org/html/2502.15524v2#S2.T1 \"Table 1 ‣ 2.2 Serverless LLM Serving in economical instances impose network bandwidth limitations that significantly increase cold start latency.\nWhile a network-only upgrade is less expensive,\nthe additional network bandwidth would be severely underutilized, as it is only used during cold starts for model fetching.\nThis paper aims to address this trade-off by minimizing cold start latency while preserving the cost benefits of resource-constrained servers.\n\nReport issue for preceding element\n\nBreakdown of LLM cold starts.\nDespite the potential benefits of serverless LLM serving,\nexisting systems suffer from propose pipeline consolidation that merges pipeline parallelism groups into individual workers to obtain optimal performance.\nSpecifically, after the parameter manager has loaded the requested model parameters in pipeline-parallel inference,\nwe inform it to continue loading the remaining part of model in background.\nOnce loading completes, the worker returns to the non-parallelized setup and serves subsequent requests with the whole model in local GPU, thereby achieving optimal performance.\nThe second part of model is loaded in low-priority CUDA streams,\nso that the performance of inference task will not be affected.\n\nReport issue for preceding element\n\n### 6.1 Worker Scaling\n\nReport issue for preceding element\n\nHydraServe provides two scaling choices.\nFirst, after workers have loaded corresponding parts of models,\nwe can perform _scaling down_ by allowing only one of them to fetch the unloaded model parts in background.\nOnce all parameters are loaded, we migrate all existing requests to that worker along with their key-value cache.\nFinally, the worker continues to generate tokens with whole model while other workers are terminated.\nFigure [4](https://arxiv.org/html/2502.15524v2#S3.F4 \"Figure 4 ‣ 3 HydraServe Overview ‣ HydraServe: Minimizing Cold Start Latency for Serverless LLM Serving in Public Clouds\")(c) demonstrates the process of scaling down.\nIn this way, we produce the first token earlier by parallelized model fetching and finally obtain a worker with whole model, similar to standard cold starts.\n\nReport issue for preceding element\n\nThe second scaling choice is _scaling up_, converting all cold-start workers into individual serving endpoints, as illustrated in Figure [4](https://arxiv.org/html/2502.15524v2#S3.F4 \"Figure 4 ‣ 3 HydraServe Overview ‣ HydraServe: Minimizing Cold Start Latency for Serverless LLM Serving in Public Clouds\")(d).\nThis approach tackles load spikes,\nwhere the autoscaler aims to deploy multiple workers for the model simultaneously.\nBy enabling the rapid creation of workers within pipeline parallelism groups,\nthis design allows the system to achieve maximum throughput more quickly.\n\nReport issue for preceding element\n\nBy default, HydraServe adopts the scaling down mechanism to reduce model fetching latency with minimal overhead.\nTo promptly react to bursty loads and transition to the scaling up mechanism, we manage worker lifecycles with a sliding window strategy.\nFor each model, the number of requests received in the previous window is recorded and used to predict the maximum number of requests likely to arrive in the next window.\nThe required number of workers is then determined based on the current waiting queue length combined with the predicted maximum number of requests expected to arrive in the next window.\nIn cold start, we create a pipeline parallelism group that is no smaller than required number of new workers and later transform the group into desired number of individual workers.\nTo handle sudden request surges, multiple pipeline parallelism groups can be created as needed.\n\nReport issue for preceding element\n\n### 6.2 Key-Value Cache Migration\n\nReport issue for preceding element\n\nAfter reducing the pipeline parallelism size,\nwe migrate all uncompleted requests,\nallowing for earlier release of resources occupied by other workers.\nSince model layers are assigned to different workers, the KV cache of these requests is distributed across workers,\nnecessitating the KV cache migration.\n\nReport issue for preceding element\n\nHydraServe performs KV cache migration inside a pipeline parallelism group, where layers are distributed across workers.\nBefore migration, we first stop scheduling of existing requests and wait for all on-the-fly batches return.\nNext, we query the cache block manager to obtain the blocks that are used by existing requests,\nand then collect these blocks from all workers with a _gather_ operation.\nBlocks are gathered to the worker with whole model and placed at different layers, according to which worker it comes from.\n\nReport issue for preceding element\n\nThe KV cache migration process in HydraServe is highly optimized.\nFirst, the whole migration workflow is performed in an individual thread and uses low-priority CUDA streams so that the inference tasks will not be affected.\nSecond, we create multiple CUDA streams when moving data from or to GPUs to utilize the GPUs’ high parallelism.\nFurthermore, the data transmission is performed in a streaming manner.\nOn the target worker, once a chunk of tensors arrives,\nit is instantly loaded to GPU in a separate CUDA stream.\nOn other workers, once a chunk of tensors is loaded from GPU to host,\nwe immediately send them to the network.\n\nReport issue for preceding element\n\n## 7 Implementation\n\nReport issue [72](https://arxiv.org/html/2502.15524v2#bib.bib72 \"\"), [73](https://arxiv.org/html/2502.15524v2#bib.bib73 \"\"), [43](https://arxiv.org/html/2502.15524v2#bib.bib43 \"\"), [33](https://arxiv.org/html/2502.15524v2#bib.bib33 \"\")\\].\nFor instance, FaaSwap \\[ [43](https://arxiv.org/html/2502.15524v2#bib.bib43 \"\")\\] and ServerlessLLM \\[ [30](https://arxiv.org/html/2502.15524v2#bib.bib30 \"\")\\] cache models in local memory or SSDs,\nwhile INFless \\[ [72](https://arxiv.org/html/2502.15524v2#bib.bib72 \"\")\\] and InstaInfer \\[ combines model caching and NPU forking to optimize cold start latencies.\nBlitzScale \\[ [31](https://arxiv.org/html/2502.15524v2#bib.bib31 \"\")\\] and λ\\\\lambdaScale \\[ [32](https://arxiv.org/html/2502.15524v2#bib.bib32 \"\")\\] achieve fast autoscaling using high-capacity networks,\nwhile Medusa \\[ [63](https://arxiv.org/html/2502.15524v2#bib.bib63 \"\")\\] speeds up CUDA graph and KV cache construction through state materialization.\n\nReport issue for preceding element\n\nHydraServe addresses the cold-start problem in public clouds.\nIts advantages are twofold.\nFirst, it imposes no requirement on the hotness of models and effectively tackles the cold-start problem without relying on caching or worker cooperation.\nSecond, it reduces the end-to-end runtime preparation overhead without any offline profiling.\n\nReport issue for preceding element\n\nLLM serving optimizations.\nMany LLM serving optimizations have been proposed to improve serving performance \\[ [51](https://arxiv.org/html/2502.15524v2#bib.bib51 \"\"), [57](https://arxiv.org/html/2502.15524v2#bib.bib57 \"\"), [74](https://arxiv.org/html/2502.15524v2#bib.bib74 \"\"), [55](https://arxiv.org/html/2502.15524v2#bib.bib55 \"\"), [75](https://arxiv.org/html/2502.15524v2#bib.bib75 \"\"), [40](https://arxiv.org/html/2502.15524v2#bib.bib40 \"\"), [76](https://arxiv.org/html/2502.15524v2#bib.bib76 \"\"), [56](https://arxiv.org/html/2502.15524v2#bib.bib56 \"\"), [77](https://arxiv.org/html/2502.15524v2#bib.bib77 \"\")\\].\nIn the area of request scheduling, Orca \\[ [75](https://arxiv.org/html/2502.15524v2#bib.bib75 \"\")\\] introduces iteration-level scheduling to run more requests in parallel,\nwhile Llumnix \\[ [56](https://arxiv.org/html/2502.15524v2#bib.bib56 \"\")\\] employs runtime scheduling to meet user SLOs.\nFor key-value cache management,\nvLLM \\[ [40](https://arxiv.org/html/2502.15524v2#bib.bib40 \"\")\\] leverages the concept of virtual memory in operating systems and InfiniGen \\[ [76](https://arxiv.org/html/2502.15524v2#bib.bib76 \"\")\\] optimizes KV block placement to improve inference efficiency.\nAdditionally, DistServe \\[ [55](https://arxiv.org/html/2502.15524v2#bib.bib55 \"\")\\] disaggregates prefill and decoding phases to mitigate their performance interference.\nHydraServe specifically focuses on reducing cold start latency in serverless LLM serving and can easily integrate these optimizations.\n\nReport issue for preceding element\n\nPipeline parallelism.\nPipeline parallelism has been widely used to scale GPU resources for training large models \\[ [45](https://arxiv.org/html/2502.15524v2#bib.bib45 \"\"), [46](https://arxiv.org/html/2502.15524v2#bib.bib46 \"\"), [47](https://arxiv.org/html/2502.15524v2#bib.bib47 \"\"), [48](https://arxiv.org/html/2502.15524v2#bib.bib48 \"\"), [49](https://arxiv.org/html/2502.15524v2#bib.bib49 \"\"), [50](https://arxiv.org/html/2502.15524v2#bib.bib50 \"\"), [58](https://arxiv.org/html/2502.15524v2#bib.bib58 \"\")\\].\nRecently, researchers have also explored its usage during inference \\[ [51](https://arxiv.org/html/2502.15524v2#bib.bib51 \"\"), [52](https://arxiv.org/html/2502.15524v2#bib.bib52 \"\"), [53](https://arxiv.org/html/2502.15524v2#bib.bib53 \"\"), [54](https://arxiv.org/html/2502.15524v2#bib.bib54 \"\"), [32](https://arxiv.org/html/2502.15524v2#bib.bib32 \"\"), [31](https://arxiv.org/html/2502.15524v2#bib.bib31 \"\")\\].\nWhile AlpaServe \\[ [51](https://arxiv.org/html/2502.15524v2#bib.bib51 \"\")\\] observes that model parallelism improves GPU utilization under load spikes and develops a model placement policy,\nHPipe \\[ [52](https://arxiv.org/html/2502.15524v2#bib.bib52 \"\")\\] and PipeEdge \\[ [54](https://arxiv.org/html/2502.15524v2#bib.bib54 \"\")\\] apply pipeline parallelism to improve the inference performance for LLMs on edge devices.\nλ\\\\lambdaScale \\[ [32](https://arxiv.org/html/2502.15524v2#bib.bib32 \"\")\\] also adopts pipeline parallelism to allow scaled-up workers to start inference earlier.\nHydraServe utilizes pipeline parallelism in a proactive manner that\ncreates multiple workers even if the model only needs one.\nIt further introduces pipeline consolidation to merge pipeline workers during inference.\n\nReport issue for preceding element\n\n## 10 Conclusion\n\nReport issue for preceding element\n\nThis paper presents HydraServe,\na serverless LLM serving system designed to reduce cold start latency in public clouds.\nHydraServe targets the most time-consuming stages of a cold start: model fetching and runtime preparation.\nTo optimize model fetching, HydraServe proactively distributes models across multiple servers,\nalleviating the burden on any single server.\nFurthermore, HydraServe incorporates pipeline consolidation that merges workers back into individual endpoints to ensure efficient resource usage and high performance for warm requests.\nFor runtime preparation, HydraServe accelerates the local worker startup by overlapping distinct stages.\nEvaluation results show that HydraServe reduces cold start latency by 1.7×\\\\times– 4.7×\\\\times compared to baselines and improves TTFT SLO attainment by 1.43×\\\\times–1.74×\\\\times under various constraints.\nBy integrating these optimizations, HydraServe offers a robust and efficient solution to meet user-defined SLOs in serverless LLM serving, especially for long-tail models.\n\nReport issue for preceding element\n\nAcknowledgments.
